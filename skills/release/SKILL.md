# Release Skill

Release notes are generated by GoReleaser from git commit messages between the
previous `v*` tag and the new release tag. Do not create per-release changelog
fragment files.

## Commit Messages

Use Conventional Commit-style subjects so GoReleaser can group release notes:

```txt
feat: add ...
fix: correct ...
docs: update ...
refactor: split ...
chore: maintain ...
```

Prefer user-facing `feat:` and `fix:` subjects when the release should clearly
show product changes. Use `chore:` for release, packaging, and maintenance work.

Configured GoReleaser groups:

- `Features`
- `Bug fixes`
- `Documentation`
- `Refactoring`
- `Maintenance`
- `Other changes`

## Release Commands

Run one of these from a clean `main` branch:

```sh
scripts/release-tag.sh patch
scripts/release-tag.sh minor
scripts/release-tag.sh major
```

The command calculates the next version from the latest `v*` tag, runs tests,
verifies npm package contents, and creates an annotated local tag.

Push the printed tag command to start the GitHub Actions release workflow.
The workflow lets GoReleaser generate GitHub Release notes from commits.

## Manual Check

To validate a specific version without tagging:

```sh
scripts/pre-release-check.sh 0.1.7
```
